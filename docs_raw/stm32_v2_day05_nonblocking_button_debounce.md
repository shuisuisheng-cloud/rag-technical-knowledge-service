# STM32F407VET6 V2 Day 5 非阻塞按键消抖

## 今日目标

今天要解决 Day 4 阻塞式消抖会暂停主循环的问题。

Day 4 检测到按键状态可能变化后，直接调用 `HAL_Delay(20)` 等待。等待期间，CPU 不能继续执行主循环中的其他功能。

Day 5 将消抖改为基于 `HAL_GetTick()` 的非阻塞方式：检测到原始电平变化后，只记录状态和变化时间，主循环继续运行；当该状态持续达到 20 ms 后，再确认按下或释放事件。

## 为什么要从 HAL_Delay() 改为 HAL_GetTick()

`HAL_Delay(20)` 会让程序停在当前代码位置约 20 ms。在这段时间内，主循环不能继续处理其他逻辑。

当前系统只有按键和 LED 时，20 ms 阻塞可能不明显。但以后加入 OLED、传感器、ESP8266 和串口通信后，频繁阻塞可能造成：

* OLED 页面刷新延迟；
* 传感器采集周期不稳定；
* 串口数据不能及时处理；
* ESP8266 返回数据处理延迟；
* 其他按键或执行器响应不及时。

`HAL_GetTick()` 只返回系统运行的毫秒数，不会暂停程序。程序可以在等待消抖时间的同时继续执行其他模块。

## Day 4 与 Day 5 的区别

### Day 4 阻塞式消抖

处理流程是：

读取按键状态
→ 发现当前状态与稳定状态不同
→ 调用 `HAL_Delay(20)`
→ 主循环暂停等待
→ 再次读取按键
→ 如果状态仍然不同，则确认按下或释放
→ 更新稳定状态

### Day 5 非阻塞式消抖

处理流程是：

读取当前按键电平
→ 与正在等待确认的状态比较
→ 如果原始电平发生变化，则保存新状态并记录 Tick
→ 主循环继续执行
→ 检查待确认状态是否持续达到 20 ms
→ 达到时间后更新稳定状态
→ 产生按下或释放事件

二者最核心的区别是：

`HAL_Delay()` 通过暂停主循环等待时间过去；`HAL_GetTick()` 通过记录时间差，使主循环在等待期间仍能继续运行。

## 核心变量

### 当前读取状态

当前读取状态由 `HAL_GPIO_ReadPin()` 在每轮主循环中获得。

它表示本轮循环真实读取到的 PA0 电平，只是瞬时原始输入，可能仍处于机械抖动期间，不能直接认定为有效按键事件。

### 上一次原始状态：`waiting`

`waiting` 保存当前正在等待确认的原始状态。

如果本轮读取状态与 `waiting` 不同，说明原始输入又发生了变化，此时需要：

* 更新 `waiting`；
* 重新记录状态变化时间；
* 重新开始消抖计时。

### 已确认稳定状态：`stable`

`stable` 保存已经连续稳定达到消抖时间，并且已经被程序确认和处理过的按键状态。

只有当 `waiting` 与 `stable` 不同，并且 `waiting` 已经持续达到 20 ms 时，才会更新 `stable`，并产生按下或释放事件。

### 状态变化时间：`tick_ms`

`tick_ms` 保存最近一次原始按键状态发生变化时的系统 Tick。

后续通过：

```c
HAL_GetTick() - tick_ms
```

计算当前待确认状态已经持续了多少毫秒。

### 消抖时间

当前消抖时间为：

```text
20 ms
```

只有某个原始状态连续保持至少 20 ms，才会被确认成新的稳定状态。

### 按下或释放事件变量

当前代码没有单独定义一个按键事件变量。

按下或释放事件是在 `stable` 更新后，通过判断新的稳定状态是有效电平还是无效电平即时产生的：

* `stable == BOARD_KEY_ACTIVE_LEVEL`：按下事件；
* `stable == BOARD_KEY_INACTIVE_LEVEL`：释放事件。

后续将按键功能封装成独立模块时，可以再设计统一的事件枚举，例如按下、释放、短按和长按。

不能只使用一个状态变量，因为程序必须区分：

* 本轮实际读取到的瞬时状态；
* 当前正在等待确认的原始状态；
* 已经经过消抖确认的稳定状态。

如果全部使用同一个变量，就无法判断某个状态是否已经持续满 20 ms，也无法避免重复触发事件。

## 核心函数

### HAL_GPIO_ReadPin()

输入是：

* GPIO 端口；
* GPIO 引脚。

本项目读取的是板载 S1 对应的 PA0。

输出类型是：

```c
GPIO_PinState
```

可能返回：

```c
GPIO_PIN_SET
GPIO_PIN_RESET
```

S1 使用下拉输入并且高电平有效，因此：

* 松开按键：`GPIO_PIN_RESET`；
* 按下按键：`GPIO_PIN_SET`。

### HAL_GetTick()

`HAL_GetTick()` 返回系统启动后经过的毫秒数，返回类型通常为 `uint32_t`。

它适合计算经过时间，因为可以保存某次状态变化时的 Tick，再用当前 Tick 减去之前的 Tick：

```c
HAL_GetTick() - tick_ms
```

如果结果大于或等于 20，就说明当前状态已经持续了至少 20 ms。

它只读取时间，不会像 `HAL_Delay()` 一样阻塞主循环。

### HAL_GPIO_TogglePin()

`HAL_GPIO_TogglePin()` 只在新的稳定状态被确认成按下状态时调用。

也就是说：

* 确认按下：翻转 D2；
* 保持按住：不重复翻转；
* 确认释放：只输出释放日志，不翻转 D2。

## 完整控制流

主循环读取当前 PA0 电平
→ 将当前电平与 `waiting` 比较
→ 如果两者不同，说明原始电平发生新变化
→ 更新 `waiting`
→ 使用 `HAL_GetTick()` 更新 `tick_ms`
→ 主循环继续运行
→ 判断 `waiting` 是否与 `stable` 不同
→ 判断 `HAL_GetTick() - tick_ms` 是否达到 20 ms
→ 达到消抖时间后，将 `waiting` 写入 `stable`
→ 如果 `stable` 是有效电平，确认按下事件
→ 输出 `KEY PRESSED` 并翻转 D2
→ 如果 `stable` 是无效电平，确认释放事件
→ 输出 `KEY RELEASED`

核心原则是：

原始状态每变化一次，就重新开始计时；只有该状态连续稳定保持满 20 ms，才会被确认成有效事件。

## 状态变化检测与软件消抖的区别

状态变化检测解决的是：

按键保持按住时，不能在每轮主循环中重复执行按下动作。

如果程序只判断当前是不是高电平，那么按住期间每轮循环都会认为按键处于按下状态，可能不断打印日志或翻转 LED。

软件消抖解决的是：

机械按键在按下或释放瞬间可能出现多次快速电平变化，不能把一次物理操作识别成多次事件。

二者不能混为一谈：

* 状态变化检测处理稳定状态是否发生改变；
* 软件消抖判断原始电平变化是否足够稳定、是否可信。

只有同时完成这两个步骤，才能得到可靠且不会重复触发的按键事件。

## 输入与输出

输入：

* S1 对应 PA0 的当前 GPIO 电平；
* 板级配置中的按键有效电平和无效电平；
* `HAL_GetTick()` 返回的系统时间；
* 20 ms 消抖时间。

输出：

* 确认按下后输出 `KEY PRESSED`；
* 确认释放后输出 `KEY RELEASED`；
* 确认按下后翻转板载 D2；
* 为后续模块提供稳定的按下和释放事件。

## 真实硬件验证结果

按下一次 S1 后，原始电平由低变高。当高电平持续达到约 20 ms 后，程序确认按下事件，串口输出：

```text
KEY PRESSED
```

同时板载 D2 翻转一次。

保持按住时，不会重复输出按下日志，D2 也不会重复翻转。

松开 S1 后，原始电平由高变低。当低电平持续达到约 20 ms 后，程序确认释放事件，串口输出：

```text
KEY RELEASED
```

释放时 D2 不发生翻转。

编译结果为：

```text
0 Error
0 Warning
```

当前功能已经通过 STM32F407VET6 核心板真实硬件验证。

## 今日遇到的问题

非阻塞消抖比阻塞式消抖更难理解，因为代码不会停在原地等待，而是依靠多个变量保存跨越多轮主循环的状态。

主要难点是区分：

* 当前实时读取状态；
* 正在等待确认的状态；
* 已经确认的稳定状态；
* 原始状态开始保持的时间。

如果这些变量职责混淆，可能导致：

* 按下一次触发多次；
* 消抖计时不断重置；
* 释放事件无法产生；
* 程序启动后错误地产生按键事件；
* 状态已经确认后仍然反复处理。

## 常见错误

### 为什么不能在每轮循环中重新初始化计时变量？

如果每轮循环都重新初始化 `tick_ms`，就无法保存按键原始状态真正发生变化的时间。

这样时间差可能永远达不到 20 ms，或者每轮都从错误的时间重新计算，消抖逻辑无法正常工作。

### 为什么不能只判断当前电平是否为有效电平？

如果只判断当前电平是否为有效电平，按键保持按住期间每轮循环都会执行按下逻辑。

必须比较稳定状态是否发生变化，才能保证一次按下只产生一次事件。

### 为什么确认事件后必须更新稳定状态？

如果确认事件后不更新 `stable`，那么 `waiting` 与 `stable` 会一直不同。

程序会在后续循环中不断满足确认条件，重复输出同一个事件。

### Tick 计数溢出时，时间差计算应注意什么？

更稳妥的写法是：

```c
HAL_GetTick() - tick_ms >= debounce_ms
```

而不是：

```c
HAL_GetTick() >= tick_ms + debounce_ms
```

由于 Tick 通常使用无符号整数，在正常回绕情况下，使用无符号减法仍可以得到正确的经过时间。

## 在 EdgeAIoT 系统中的意义

非阻塞按键事件以后可以用于：

* OLED 页面切换；
* 正常、外出、睡眠等工作模式切换；
* 报警确认；
* 执行器本地控制；
* 网络异常时的离线操作；
* 系统配置和调试入口。

它与其他模块的关系是：

```text
物理按键
→ GPIO 输入读取
→ 非阻塞消抖
→ 按下、释放、短按或长按事件
→ OLED 页面、设备模式和执行器控制
```

非阻塞设计使按键等待消抖期间，主循环仍然能够继续处理 OLED、传感器和串口通信。

## 面试时怎么讲

Day 4 使用 `HAL_Delay(20)` 实现了基础消抖，但这种方式会阻塞主循环，不适合后续同时处理 OLED、传感器和串口数据。

因此 Day 5 使用 `HAL_GetTick()` 将按键消抖重构为非阻塞状态机。

程序分别保存实时读取状态、正在等待确认的 `waiting`、已经确认的 `stable`，以及状态变化时间 `tick_ms`。

原始电平每变化一次，就更新待确认状态并重新计时；只有该状态连续保持满 20 ms，才会更新稳定状态并产生按下或释放事件。

确认按下时翻转板载 D2，保持按住不会重复触发，确认释放时只输出释放日志。

该功能已经在 STM32F407VET6 核心板上完成真实硬件验证，但目前还没有扩展到短按、长按和独立按键模块。

## 如果从零重写，第一步做什么

第一步先确定按键的硬件特性：

* 使用哪个 GPIO；
* 按下时是高电平还是低电平；
* 是否需要上拉或下拉；
* 消抖时间是多少。

然后设计状态变量：

* 当前读取状态；
* 待确认状态；
* 已确认稳定状态；
* 状态变化时间；
* 消抖时间；
* 按下和释放事件。

之后按照以下顺序实现：

```text
验证 GPIO 输入
→ 检测原始状态变化
→ 保存新状态和 Tick
→ 判断状态持续时间
→ 更新稳定状态
→ 输出按下和释放事件
→ 最后连接 LED 等业务逻辑
```
